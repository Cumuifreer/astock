# 指标规则构建器重做设计

日期：2026-05-26

## 背景

当前“指标库 + 信号模式 + 策略页”的方向是对的，但交互和执行语义还没有统一：

- 策略页让用户给指标选择“过滤 / 加分 / 风控 / 展示”，这是工程内部角色，不是用户能直接理解的筛选语言。
- 一些“观察可用”的指标，如主力净额、龙虎榜净额、筹码胜率，数据已经能在数据仓库或个股档案里看到，但没有真正进入分析引擎。用户把它们设成“加分”时，实际不会影响分析结果。
- 题材热度、RPS、量比、筹码胜率等指标缺少单位、取值范围、推荐阈值和当前市场分布提示，用户不知道该填多少。
- 目前的策略参数仍以 `min_rps120`、`min_topic_heat` 等散落字段为主，新指标要能参与筛选通常需要写一套专门参数和后端逻辑。

本次重做目标是把策略编辑器从“角色下拉框”升级为“可执行指标规则构建器”，接近 TradingView Screener 的心智：选择指标，选择条件，填写阈值，决定是否筛选、加分、风控或仅展示。

## 目标

1. 让用户看到的每个可选指标都有明确用途：可筛选、可评分、可风控、可展示。
2. 所有指标编辑都使用直观规则：`指标 + 操作符 + 数值/范围 + 动作`。
3. 每个指标显示单位、取值解释、推荐阈值和最新市场分布。
4. “信号模式”成为规则模板，用户可以增删改规则，不再面对内部字段名和散落参数。
5. 后端提供统一规则执行器，逐步替代大量硬编码 `min_*` 参数。
6. 保持旧策略可运行，并提供兼容迁移。

## 非目标

- 不在第一版引入完整公式编辑器或 Pine Script 式脚本语言。
- 不在第一版做多周期分钟级指标。盘中雷达继续沿用当前日线/快照能力。
- 不一次性接入所有 Tushare 接口。只有已经进入分析帧的字段能标记为可执行；已入库但尚未进入分析帧的字段只能标记为仅展示。
- 不移除现有系统策略的运行逻辑。先兼容，再逐步改为规则驱动。

## 核心概念

### 指标

指标是用户能理解的市场变量，例如：

- RPS120
- 题材热度
- 主力净额
- 龙虎榜净额
- 筹码胜率
- 平台振幅
- 突破上沿距离

指标必须有元数据：

```ts
type IndicatorDefinition = {
  id: string;
  name: string;
  category_id: string;
  source: string;
  formula: string;
  description: string;
  value_type: "number" | "money" | "percent" | "ratio" | "score" | "boolean" | "event";
  unit: string;
  range_hint?: { min?: number; max?: number; label?: string };
  direction: "higher_better" | "lower_better" | "range_better" | "neutral";
  precision: number;
  supported_actions: Array<"filter" | "score" | "risk" | "display">;
  supported_operators: string[];
  default_operator: string;
  recommended_rules: RecommendedRule[];
  data_status: "executable" | "display_only" | "planned";
  analysis_field?: string;
  missing_policy_options: Array<"skip" | "keep" | "neutral">;
};
```

### 规则

规则是用户真正编辑的东西：

```ts
type StrategyRule = {
  id: string;
  indicator_id: string;
  action: "filter" | "score" | "risk" | "display";
  operator: "gte" | "lte" | "gt" | "lt" | "between" | "eq" | "neq" | "is_true" | "recent";
  value?: number | string | boolean;
  value2?: number | string;
  window_days?: number;
  weight?: number;
  missing_policy: "skip" | "keep" | "neutral";
  enabled: boolean;
};
```

用户不再选择“角色”。用户添加指标时，界面直接创建一条规则。动作以按钮或标签呈现，而不是下拉框。

### 信号模式

信号模式是规则模板，不是基础引擎。突破回踩、平台突破、平台临界、趋势共振、题材共振突破都是信号模式。用户可以改名、备注、增删指标规则和组合条件。

新建信号模式默认只包含基础股票池规则：

- 最低股价
- 成交额门槛
- 流通市值范围
- 是否包含北交所
- 是否排除科创板
- 缺失数据处理

其他规则由用户从指标库添加。

## 用户体验设计

### 策略页结构

策略编辑页改成三段：

1. 策略头部：策略名称、信号模式、运行方式、保存/运行按钮。
2. 规则工作台：以规则卡片为核心，按分类分组。
3. 结果解释：最近一次运行后展示每条规则影响了多少股票、哪些规则最严格、哪些指标缺失。

### 指标添加

“加入指标”打开一个指标抽屉，而不是一个很长的下拉框。

抽屉左侧是分类：

- 基础股票池
- 基础行情
- 技术强弱
- 平台形态
- 趋势共振
- 资金流向
- 题材行业
- 事件异动
- 筹码成本
- 风险过滤
- 市场环境

右侧是指标列表。每个指标显示：

- 名称
- 当前能力：可筛选、可评分、可风控、仅展示、待接入
- 单位和取值范围
- 推荐规则按钮

示例：

```text
题材热度
0-100 分，越高越强
推荐：>= 65 强势，>= 80 极强
[筛选] [加分] [展示]
```

### 规则卡片

规则卡片显示为一句可读条件：

```text
题材热度  >=  65    筛选
```

或：

```text
主力净额  >  0 元    加分，权重 8
```

卡片中必须包含：

- 指标名称
- 操作符：高于、低于、介于、等于、最近 N 日出现
- 数值输入，带单位
- 动作按钮：筛选、加分、风控、展示
- 缺失处理：跳过、保留、中性
- 指标解释与推荐阈值入口

不适用的动作不能出现。例如 `龙虎榜净额` 在没有进入分析帧前只能选择“展示”，不能假装“加分”。

### 指标解释

用户点开指标说明时看到：

- 这是什么：简短解释
- 怎么算：公式或口径
- 单位：元、%、倍、0-100 分、次数
- 当前覆盖：例如 `5,189 / 5,527`
- 最新分布：P25、P50、P75、P90
- 推荐阈值：保守、标准、激进
- 数据来源和最新日期
- 参与方式：可筛选、可加分、可风控、可展示

题材热度示例：

```text
题材热度
单位：0-100 分，越高代表所属题材越活跃。
口径：题材内上涨比例、RPS 高分股比例、涨停数量、平均涨幅综合计算。
当前市场：P50 42，P75 63，P90 78。
建议：>= 65 代表强势题材，>= 80 代表极强题材。
```

### 运行反馈

分析完成后，策略页和分析结果页都要解释规则效果：

```text
题材热度 >= 65：5527 -> 842，移除 4685
主力净额 > 0：842 -> 361，加分规则命中 361
龙虎榜净额 最近3日 > 3000万：数据覆盖 151，只作为展示列
```

这能让用户知道每条规则到底有没有生效。

## 后端设计

### 分析帧

分析服务先构造统一 `analysis_frame`。每个可执行指标必须有对应字段。字段按实施阶段进入分析帧，不能先在 UI 里承诺可筛选或可加分。

阶段二先纳入当前已经稳定参与分析的字段：

- 基础行情：最新价、成交额、涨跌幅、换手率、量比、流通市值、振幅
- 技术强弱：RPS20、RPS60、RPS120、均线距离、MACD 状态
- 平台形态：平台振幅、突破上沿距离、距平台上沿、突破量比
- 题材行业：题材数、题材热度、题材涨停数
- 风险过滤：ST、停牌、过热风险

阶段三再纳入新增数据字段：

- 资金流向：主力净额、资金净流入
- 事件异动：涨跌停状态、龙虎榜净额、龙虎榜最近出现天数
- 筹码成本：筹码胜率、中位成本、当前价相对中位成本距离

市场环境如果没有指数权限，先作为整体上下文展示，不作为个股规则筛选。

### 规则执行器

新增统一执行函数：

```py
apply_rule(frame, rule, indicator_definition) -> RuleResult
```

规则动作：

- `filter`：不满足条件则剔除。
- `score`：满足条件则加分，不满足不剔除。
- `risk`：满足风险条件则扣分或标记。
- `display`：只加入结果列，不影响候选。

每条规则返回：

- before_count
- after_count
- matched_count
- missing_count
- effect_summary

### 兼容旧策略

旧策略仍保留 `StrategyConfig` 的字段。保存或加载时增加转换层：

- `min_price` 转成 `latest_price >= min_price`
- `min_amount` 转成 `amount >= min_amount`
- `min_rps120` 转成 `rps120 >= min_rps120`
- `min_topic_heat` 转成 `topic_heat >= min_topic_heat`
- `max_turnover` 转成 `turnover_rate <= max_turnover`
- 平台/趋势的复杂参数先保留现有专用逻辑，同时在 UI 中以规则卡片展示。

这样用户旧策略不丢，后端也不需要一次性重写所有形态算法。

## 指标能力分级

用户能看到的指标分三类：

### 可执行

已经进入 `analysis_frame`，可以筛选、加分、风控或展示。

示例：

- 最新价
- 成交额
- 换手率
- 量比
- RPS20/60/120
- 平台振幅
- 题材热度
- 题材涨停数
- 主力净额，完成阶段三后
- 龙虎榜净额，完成阶段三后
- 筹码胜率，完成阶段三后

### 仅展示

有数据，但暂时没有稳定规则口径，或覆盖太低，不建议参与筛选。

示例：

- 市场环境宽度，在指数权限不完整时只做上下文。
- 游资席位名称，在明细覆盖不足时只展示。

### 待接入

指标库可以展示说明，但不能加入策略。待接入项不出现在“加入指标”抽屉的可选规则列表里。

## 指标适配建议

下表是最终规则构建器要支持的指标口径。阶段二先支持当前分析帧已有字段；阶段三再把资金、事件和筹码字段从仅展示升级为可执行。

| 指标 | 单位 | 推荐操作符 | 推荐规则 |
| --- | --- | --- | --- |
| 最新价 | 元 | >=, between | >= 4 |
| 成交额 | 元 | >= | >= 8000 万 |
| 换手率 | % | between, <= | 3%-12% |
| 量比 | 倍 | >= | >= 1.5，突破可 >= 2.5 |
| RPS20/60/120 | 分 | >= | >= 70，强势 >= 80 |
| 平台振幅 | % | <= | <= 12% |
| 突破上沿距离 | % | between | 3%-8% |
| 题材热度 | 0-100 分 | >= | >= 65，极强 >= 80 |
| 题材涨停数 | 家 | >= | >= 1 或 >= 2 |
| 主力净额 | 元 | >, >= | > 0；强势 > 3000 万 |
| 资金净流入 | 元 | >, >= | > 0 |
| 龙虎榜净额 | 元 | >, >=, recent | 最近 5 日 > 3000 万 |
| 筹码胜率 | % | >=, between | >= 60% 或 40%-80% |
| 中位成本距离 | % | between, <= | 当前价距中位成本 <= 15% |
| 涨停事件 | 事件 | recent, eq | 最近 3 日出现涨停 |

## UI 细节

- 不再使用原始 `<select>` 下拉框选择“过滤/加分/风控/展示”。
- 用分段按钮显示动作：筛选、加分、风控、展示。
- 操作符使用短按钮：高于、低于、介于、等于、最近出现。
- 数值输入右侧固定显示单位。
- 推荐阈值做成可点击小按钮：保守、标准、激进。
- 对覆盖低的指标显示覆盖提示，不允许默认作为强筛选。
- 规则卡片要保持紧凑，适合反复编辑，避免大段说明挤占空间。
- 说明内容放在可展开详情或侧栏，不直接堆在规则卡片上。

## 数据地图和指标库同步

数据地图继续负责“数据是否新鲜、覆盖是否够”。指标库负责“指标如何使用”。

两者通过 `indicator_id` 对齐：

- 数据地图显示数据表覆盖和最新日期。
- 指标库显示指标单位、可用动作和推荐规则。
- 策略页只允许加入 `data_status != planned` 的指标。

## 错误处理

- 如果规则引用的指标缺少字段，保存时提示“该指标暂不能执行”。
- 如果运行时字段覆盖过低，分析结果里显示规则覆盖率。
- 如果所有候选被某条规则过滤掉，零结果原因指向具体规则。
- 如果旧策略加载失败，回退到旧 `StrategyConfig` 专用表单并给出提示。

## 测试计划

后端：

- 指标元数据完整性测试：每个可执行指标必须有单位、操作符、字段名和推荐规则。
- 规则执行器测试：gte、lte、between、recent、missing_policy。
- 旧策略迁移测试：旧 `min_*` 参数转成等价规则。
- 分析结果解释测试：每条规则返回 before/after/matched/missing。

前端：

- 指标抽屉分类、搜索、推荐阈值。
- 规则卡片编辑：动作、操作符、数值、缺失处理。
- 不可执行指标不能选择筛选/加分/风控。
- 保存后重新加载规则不丢失。
- 移动端规则卡片不横向溢出。

端到端：

- 创建“题材热度 >= 65 + RPS120 >= 80”的策略并运行。
- 创建“主力净额 > 0 加分”的策略并验证分数变化。
- 创建“龙虎榜净额仅展示”的策略并确认结果列出现但不影响过滤。

## 实施顺序

### 阶段一：元数据和前端规则编辑器

- 扩展指标定义字段。
- 策略页改为规则卡片和指标抽屉。
- 去掉角色下拉框。
- 旧信号模式转换为规则卡片展示。

### 阶段二：通用规则执行器

- 新增 `strategy_rules` 结构。
- 旧参数转换为规则。
- 基础行情、RPS、题材指标先走通用执行器。

### 阶段三：扩展资金、事件、筹码执行能力

- 主力净额、资金净流入进入分析帧。
- 龙虎榜净额和最近出现天数进入分析帧。
- 筹码胜率、中位成本距离进入分析帧。
- 更新分析结果解释和候选详情。

### 阶段四：规则反馈和推荐阈值

- 每个指标生成最新市场分布。
- UI 显示 P50/P75/P90 和推荐阈值。
- 运行结果显示每条规则影响。

## 验收标准

- 用户不再看到“过滤 / 加分 / 风控 / 展示”的裸下拉框。
- 每个可加入策略的指标都能解释它的单位、取值范围和推荐阈值。
- 如果指标不能实际参与筛选或评分，UI 不允许用户把它设置为筛选/加分。
- 主力净额、龙虎榜净额、筹码胜率这些新增数据要么真实参与分析，要么只能作为展示列，不能出现“看起来能用但实际没用”的状态。
- 运行完成后，用户能看到每条规则对候选数量和评分的影响。
- 旧策略仍能加载、保存和运行。
