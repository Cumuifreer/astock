# 非活跃股票标签与数据地图口径设计

日期：2026-05-27

## 背景

数据地图目前把 `stock_basic` 的全部股票都作为股票型数据能力的分母。这个口径会把已经没有可维护行情的数据也算进覆盖缺口，例如 `000003.SZ`、`000005.SZ`。本地 DuckDB 检查显示：

- `stock_basic` 总数为 5,527。
- 其中 `suspended = TRUE` 的股票为 324。
- `000003.SZ`、`000005.SZ` 都在 `stock_basic` 中，且 `suspended = TRUE`，历史 K 线为 0。
- 数据地图的 `5,205 / 5,527`、`5,211 / 5,527` 这类差额，和这批非活跃股票的数量级一致。

这不是单纯的补齐失败，而是股票生命周期状态没有进入数据质量口径。

## 目标

1. 给已经不进入日常数据维护的股票打统一标签：`非活跃`。
2. 数据地图的股票型覆盖率使用活跃股票池作为分母，不让非活跃股票制造虚假缺口。
3. 数据仓库仍然保留非活跃股票，可搜索、可查看、可解释，不从数据库里删除。
4. 更新、补齐和策略分析默认围绕活跃股票运行，避免持续请求已无行情股票。
5. 前端用清晰、克制的方式说明统计口径，例如“已忽略 324 只非活跃股票”。

## 非目标

- 不删除 `stock_basic` 中的非活跃股票。
- 不在第一版区分退市、吸收合并、长期停牌、代码变更等细分原因。
- 不把个股某一天 `tradestatus = 0` 的正常停牌直接等同于非活跃。
- 不改变分析“只读本地数据库”的原则。
- 不引入登录、权限或复杂管理后台。

## 术语

### 活跃股票

日常数据地图、补齐任务、策略分析应该默认覆盖的股票。初版口径为：

```sql
stock_basic.suspended IS DISTINCT FROM TRUE
```

### 非活跃股票

已不应进入日常覆盖率分母和默认补齐任务的股票。初版由 `stock_basic.suspended = TRUE` 映射为前端标签 `非活跃`。

`非活跃` 是产品语言，不承诺具体法律状态。它可以覆盖退市、合并、代码不再维护、长期无可用行情等情况。

### 当日停牌

当日停牌仍由历史 K 线或行情数据中的 `tradestatus` 表达。它是交易日状态，不等于股票生命周期状态。

## 后端设计

### 股票池口径 helper

在 `DataService` 和 `UpdateService` 中集中使用同一套活跃股票 SQL 条件，避免不同能力各写各的判断：

```sql
b.suspended IS DISTINCT FROM TRUE
```

建议封装为常量或 helper，例如：

- `ACTIVE_STOCK_WHERE = "b.suspended IS DISTINCT FROM TRUE"`
- `active_stock_count()`
- `inactive_stock_count()`

### 数据地图分母

`DataService.refresh_capabilities()` 中的股票型能力分母从全部股票数改为活跃股票数：

- 历史 K 线
- 当天行情快照
- 股票基础信息
- 流通市值
- 换手率
- RPS
- 振幅
- ST / 停牌状态
- 每日指标
- 技术因子
- 资金流向
- 筹码分布
- 概念 / 行业成分

事件型和数据集型口径保持原样：

- 涨跌停：事件数，不用股票分母。
- 龙虎榜 / 游资：事件数，不用股票分母。
- 市场环境：日期数，不用股票分母。

覆盖计数也应尽量只统计活跃股票，避免非活跃股票旧数据把覆盖率抬高。

### Overview 统计

`overview.stock_count` 继续保留总股票数容易造成误解。建议新增：

- `stock_count`：全部股票数，保留兼容。
- `active_stock_count`：活跃股票数。
- `inactive_stock_count`：非活跃股票数。

首页“股票池”卡片优先展示活跃股票数，副文案显示“忽略 X 只非活跃”。

### 数据仓库列表

`list_stocks()` 增加状态筛选参数：

- `status=active`：默认，只显示活跃股票。
- `status=inactive`：只显示非活跃股票。
- `status=all`：显示全部。

普通浏览默认遵守状态筛选。搜索框输入完整股票代码时，如果活跃结果为空但存在非活跃匹配，前端应显示该非活跃股票并带 `非活跃` 标签，避免用户觉得数据被删除。

返回字段新增或复用：

- `is_active`
- `status_label`

其中 `status_label` 对非活跃股票返回 `非活跃`，前端无需重复推断。

### 更新与补齐任务

完整更新和轻量更新应默认跳过非活跃股票。当前部分 Tushare 增强补齐已经用 `b.suspended IS DISTINCT FROM TRUE`，需要统一检查历史 K 线、快照补齐、能力补齐路径。

如果未来需要维护历史档案，可以单独提供“包含非活跃”的维护选项；第一版不做。

## 前端设计

### 数据地图

数据账本顶部增加一行口径说明：

```text
按活跃股票统计 · 已忽略 324 只非活跃股票
```

这行不做警告样式，只作为统计说明。数据地图的缺口状态应只反映活跃股票的数据质量。

股票型覆盖率继续显示 `覆盖数 / 活跃分母`。例如原来 `5,205 / 5,527`，改为接近 `5,203 / 5,203` 或实际活跃覆盖数。

### 数据仓库

股票名称旁增加紧凑状态标签：

- 活跃股票不显示标签，保持表格安静。
- 非活跃股票显示 `非活跃` 标签。

筛选区增加状态筛选：

- 活跃
- 非活跃
- 全部

默认选中“活跃”。这个控件放在交易所 / 板块筛选附近，使用现有 filter chip 样式。

### 视觉语气

`非活跃` 标签应是低饱和、低优先级状态，不用危险红。建议使用灰绿或琥珀灰，让它像数据状态而不是错误。

## 测试

后端测试：

1. `refresh_capabilities()` 使用活跃股票分母，非活跃股票不增加 `missing_count`。
2. 非活跃股票有旧历史数据时，不应抬高股票型覆盖数。
3. `overview()` 返回 `active_stock_count` 和 `inactive_stock_count`。
4. `list_stocks(status=active|inactive|all)` 正确筛选。
5. 能力补齐路径默认不选择非活跃股票。

前端验证：

1. 数据地图显示活跃口径说明。
2. 数据仓库默认只显示活跃股票。
3. 非活跃筛选能看到 `000003.SZ`、`000005.SZ` 这类股票，并显示 `非活跃` 标签。
4. 在默认活跃筛选下搜索完整非活跃代码时，仍能看到对应股票和 `非活跃` 标签。
5. `npm run build` 通过。

## 部署与回滚

本次不删除数据，迁移风险低。部署仍按当前工作流：

1. 本地实现、测试、提交、推送 `main`。
2. 云服务器 `git pull`。
3. 安装依赖并构建前端。
4. 运行 `python scripts/init_db.py`。
5. 重启 systemd 服务。

如果需要回滚，只要回滚代码即可；数据库里原有 `stock_basic.suspended` 字段和股票数据不需要恢复。
